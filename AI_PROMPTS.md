# AI_PROMPTS.md — AI 协同研发报告

## 0. 协同研发方法论（本项目核心做法）

本项目我没有把它当成"让一个 AI 把代码生成出来"，而是当成一次**多 AI 协作的工程交付**来组织。我担任总设计与对接角色，协调两个 AI 并行开发，自己负责定标准、审一致性、做端到端验收和拍板争议点。

**分工：**
- **VS Code + Claude**：后端、无人机模拟端、启动脚本、Docker、后端文档。
- **Codex（GPT-5）**：前端三页、Leaflet 地图、UI 与交互。
- **我（总协调）**：接口契约设计、阶段任务拆解、字段一致性审查、端到端验收、有争议改动的最终拍板。

为规避多 AI 协作最大的三个风险——接口对不上、互相覆盖、风格漂移——我立了三条工程纪律：

1. **接口契约先行（唯一事实来源）**：动手写代码前先冻结 `API_CONTRACT.md v1.0`，把 9 个接口的路径、字段名、枚举、告警规则全部定死（统一 snake_case）。两个 AI 都以它为唯一标准实现；任何改动必须先记入 CHANGELOG 并经我确认。这让前端可以先用 mock 数据独立开发、后端独立实现，最后联调几乎零返工。

2. **项目纪要驱动（阶段化推进 + 过程留痕）**：两个 AI 共享一份 `PROJECT_LOG.md`，每完成一个阶段就追加各自的"完成内容 / 是否改接口 / 自测结果 / 遗留问题"，且只许在自己段落追加、不许覆盖对方记录。我据此判断进度、发现风险、派发下一阶段任务，形成"派发 → 产出 → 复盘 → 再派发"的闭环。

3. **人工把控边界与验收**：AI 负责实现与自测，但不允许越界改对方目录、不允许编造未完成的内容。我对关键节点亲自验收——例如本机双击 `start.bat` 实测一键启动、逐条核对文档与代码一致性、对照原始题目检查交付物是否齐全。

**典型决策实例：**
- 守住契约：异常分支"未知 alert_id 返回 404"原契约未规定，判定为健壮性加固而非违约，记入 CHANGELOG 并知会前端，确保不破坏既有对接。
- 交付可靠性优先：前端选用零依赖原生技术栈（不引入 npm/构建），并为地图设计了不依赖网络的"示意沙盘"离线兜底模式，确保评分方在任意环境下双击即可运行、断网也能完整演示。
- 真实踩坑闭环：start.bat 一键启动曾因"前端端口未就绪就打开浏览器"导致打不开，定位为时序问题后改用端口就绪探测修复（详见后端记录）。

**这套方法的价值**：它把"AI 写代码"升级为"用工程流程编排多 AI 协作"——需求拆解、契约对齐、并行开发、联调验收、过程留痕，每一步都可追溯、可交接，可直接复用到真实的 AI 应用落地与交付项目中。

## 3. 后端开发记录

### 3.1 由 AI 辅助完成的模块
- `backend/app.py`：契约全部 9 个接口、告警规则引擎（high/medium/low + 同区域连续异常≥3 升 high）、基于“最近心跳读时计算”的在线/离线判定。
- `backend/config.py`：端口、心跳超时、模拟数量、上报/心跳间隔、各级告警阈值全部抽成配置项。
- `drone-simulator/simulator.py`：自动注册 N 架、定时心跳+巡检上报、位置漂移成轨迹、按概率制造异常值触发三档告警。
- `start.bat` / `requirements.txt`：一键启动与依赖锁版本。
- `ARCHITECTURE.md` / `README.md`：架构拓扑+核心流程+扩展性、安装部署说明。

### 3.2 关键 Prompt / 方法
- “动手前先完整读 API_CONTRACT.md，接口路径/字段/枚举/告警规则一字不改，全 snake_case” —— 把契约作为硬约束写进每次任务，保证零改动对接。
- “阈值、超时、端口、模拟数量全部做成 config 配置项” —— 强制可调性，便于演示时改参数。
- “每完成一个阶段把进度追加到 PROJECT_LOG.md，不覆盖前端纪要” —— 保证前后端共享一份可复盘记录。

### 3.3 遇到的问题与定位/修复（如实记录）
- **现象**：阶段1 自测时，用 curl 在 Windows + bash 管道里 `POST /api/drones/register` 带中文 `name`，返回 `{"detail":"There was an error parsing the body"}`。
- **AI 定位过程**：对比同样引号风格、纯 ASCII 的 `/api/telemetry` 请求可正常解析，判断不是接口/模型问题，而是命令行管道对中文 body 的编码在传输中损坏（非 UTF-8 透传）。
- **最终修复**：用纯 ASCII 的 `name` 重测，注册、列表、轨迹全部通过；确认后端本身无 bug，无需改代码。结论：属测试工具的编码问题，已在 PROJECT_LOG 标注。
- **现象**：用 `timeout 14 python simulator.py` 抓日志时，日志文件为空。
- **定位**：被 `timeout` 强杀时 Python stdout 行缓冲未刷出所致；通过查后端 `GET /api/drones`（5 架已注册、电量在降、三档告警已生成）反证模拟端实际运行正常。
- **修复**：无需改代码，确认为采集方式问题。

### 3.4 阶段2 加固（接口零改动）
- 新增根路由 `GET /`（返回 service/status/接口清单），未知 `drone_id`/`alert_id` 统一返回 404 JSON（中英文提示），列表接口无数据返回空数组，请求体由 pydantic 校验非法参数返回 422。以上均为加固，未改动契约已有路径/字段/枚举/规则。

### 3.5 阶段5/6 增强与问题定位

#### Bug：start.bat 双击后浏览器打开了但前端页面打不开
- **现象**：用户双击 `start.bat`，浏览器确实自动弹出了，但页面打不开（无法访问 5500）；手动 `cd frontend` 再 `python -m http.server 5500` 就一切正常。
- **定位**：是典型的**启动时序问题**。脚本启动 `python -m http.server 5500` 后只 `timeout 2` 等 2 秒就 `start` 浏览器，而 http.server 进程从拉起到真正 **bind 5500 端口就绪**需要更长时间——浏览器在端口还没就绪时就发起了请求，自然打不开。手动操作之所以正常，是因为人工敲完命令再切浏览器，天然等够了时间。另外 `cd /d ... &&` 链式与 PATH 上下文也可能拖慢前端窗口启动。
- **修复**：把"固定等 2 秒"改成**就绪探测**——用 PowerShell `TcpClient` 循环探测 `127.0.0.1:5500`，**探测通了再 `start` 浏览器**（最多约 20 秒，超时也兜底打开并提示刷新）；Python 命令做 `python → py` 回退，找不到则清晰报错；后端/模拟端/前端各用 `start "标题" cmd /k` 独立窗口、全程 `pushd "%~dp0xxx"` 绝对路径，保证可从任意目录双击运行。

#### 核查题目4.1"平台地址配置"
- **现象/核查**：题目要求模拟端连接后端的平台地址必须是配置项。检查发现地址虽未散落在代码里，但缺少题目语义明确的独立配置项。
- **修复**：在 `config.py` 新增 `PLATFORM_URL`，支持环境变量覆盖（`os.environ.get("PLATFORM_URL", "http://localhost:8000")`），`simulator.py` 改读 `config.PLATFORM_URL`；Docker 等场景可传 `http://backend:8000`（服务名）。已验证默认值与环境变量覆盖均生效。

#### 核查题目3.2 可选项 image_url
- **现象/核查**：题目3.2 允许巡检上报带"图片 URL/模拟识别结果"，但模拟端原本上报 `image_url` 恒为 `None`，可选项形同虚设、与文档描述不符。
- **修复**：触发疑似火情的异常档（low/medium/high）上报时带模拟图片识别快照 URL（`https://example.com/fire-snapshot/{drone_id}-{timestamp}.jpg`），正常数据仍为 `None`。已验证异常档带 URL、正常档为 None，后端接受返回 200，与 `ARCHITECTURE.md` 2.3 描述一致。

#### 加分：Docker 一键启动
- **动机**：题目4.3 允许"一键启动脚本 **或** Docker 启动方式"，在 `start.bat` 之外补 Docker 方案作为加分。
- **实现**：`backend/` 与 `drone-simulator/` 各加 `Dockerfile`（python:3.11-slim），根目录 `docker-compose.yml` 一条 `docker-compose up` 起后端+模拟端+前端（前端用 nginx:alpine 挂载静态文件到 5500）；模拟端经 `PLATFORM_URL=http://backend:8000` 用服务名连后端，正好复用上面新增的配置项。本环境无 Docker daemon，编排逻辑已就绪，建议有 Docker 的环境 `docker-compose up --build` 终验。

### 3.6 交付体验与环境隔离的人工修正

这一节记录两个由人工实测发现、判定为"交付软件应有规范"而非功能 bug 的修正（按现象→定位→修复）。

#### 环境污染：一键脚本最初直接污染全局 Python
- **现象**：一键脚本最初直接向全局 Python 安装锁定版本依赖，实测把用户机器上原有的 `fastapi` / `pydantic` 降级，导致用户另一项目（gradio）的依赖冲突报错。
- **定位**：判定为**交付脚本的设计缺陷**——把锁版本依赖装进全局解释器，会在评分方/用户机器上改动其既有环境，这对一份要拿去运行的交付物是不可接受的副作用。
- **修复**：改为**项目级 `.venv` 虚拟环境隔离依赖**——脚本先检查/创建项目内 `.venv`，所有 `pip install` 与服务启动都用 `.venv` 里的 python，依赖只落在项目自身、不再触碰全局环境。这是可交付软件应有的环境隔离规范。

#### 启动体验：从多个黑窗 bat 改为无黑窗双击即用
- **现象**：用 bat 双击启动会弹出多个黑色终端窗口（后端/模拟端/前端各一个），且窗口长期停留，不够像成品。
- **定位**：判定为**交付体验问题**——评分人第一个动作就是双击启动，多个黑窗既不专业、也容易误关导致服务中断。
- **修复**：改用 `start.vbs` 以**隐藏窗口方式**在后台拉起服务，日志统一落到 `logs/`、各进程 PID 记录到 `logs/pids.txt` 便于 `stop.bat` 一键停止，并在后端 8000 与前端 5500 两端口都就绪后才打开浏览器，实现"双击即用、无黑窗"；保留可见模式入口（`start.bat` / `start_debug.bat`）供排错。

## 前后端联调（前端 Codex 填写）

按「现象 → 定位 → 修复」记录联调期间踩过的三个真实 Bug。

### Bug1：巡检地图底图空白 + 无人机标记破图
- **现象**：切到巡检地图页，地图区域一片空白（或只有灰底），无人机标记显示成破损图片图标，控制台报瓦片和 marker 图片加载失败。
- **定位**：底图用的是 OpenStreetMap 瓦片，国内网络基本拉不出来；同时 Leaflet 默认 marker 图标依赖其 CDN 上的 `marker-icon.png / marker-shadow.png`，该路径在本地静态服务下也取不到，导致标记破图。两者都是"依赖外部不可达资源"。
- **修复**：底图换成高德免 key 瓦片（`webst0{s}.is.autonavi.com/appmaptile?style=6...`）并叠加中文路网注记层；无人机标记全部改为 `L.divIcon` + CSS 自绘圆点，按状态配色（online 绿 / cruising 蓝 / returning 橙 / offline 灰），疑似火点改为 CSS 红色脉冲圆点。彻底不再依赖任何外部图片。

### Bug2：地图瓦片只铺中间一小簇、错位且有约 512px 缝隙
- **现象**：地图能出来但只在中间挤了一小簇瓦片，四周黑，瓦片之间有明显错位和大缝隙；从别的页签切过来尤其明显。
- **定位**：地图是在 `display:none` 的隐藏标签页里 `initMap` 的——容器此时宽高被算成 0，Leaflet 按 0 尺寸初始化，后续即使显示出来也不会自动重算，瓦片定位全乱。缝隙则是 leaflet.css 未生效时瓦片 `img` 退回普通文档流（非 absolute 定位）所致。
- **修复**：改为"首次切到巡检地图标签、容器真正可见后"才 `ensureMap/initMap`；切页后延迟执行 `invalidateSize()` 并用当前 center/zoom `setView()` 强制重算铺满；加 `window resize` 监听在地图可见时 `repairMapLayout`；并补一段 Leaflet 关键布局 CSS 兜底（pane/tile/marker/overlay 强制 `position:absolute`、瓦片强制 256×256），防止 css 加载慢时错位。

### Bug3：Real 模式偶发 "Failed to fetch" 红字一直挂着
- **现象**：切到 Real 真实后端模式，页面偶尔蹦出红色 `Failed to fetch` 错误条，而且即使下一次轮询已经成功，这条错误也不消失，一直挂在顶部。
- **定位**：2 秒轮询里某次请求偶发失败（网络抖动/后端瞬时忙）会渲染错误提示，但提示的清除只在"出错时覆盖"，没有在"请求成功"时主动清掉；加之没有超时，慢请求会堆积放大偶发失败的观感。
- **修复**：请求成功即立刻清除错误提示；单次失败自动重试一次再报错；给 `fetch` 加 8 秒超时（AbortController）；失败时保留上一次的数据，不清空页面，避免画面闪烁空白。改完后偶发抖动不再产生常驻红字。

## 6. 人工判断与修正

- **坚持契约零改动**：阶段2 加固时，对“未知 alert_id 是否该返回 404”做了判断——原契约未规定该场景行为，改 404 属加固而非违约，并在日志/CHANGELOG 标注，确保不破坏前端既有对接。
- **不越界**：明确后端不碰 `frontend/` 源码，`start.bat` 仅“引用”前端目录启动静态服务；前端相关文档章节一律留占位，不编造。
- **核对题目与契约冲突**：人工要求先比对原始场景题与 API_CONTRACT.md，确认中文状态/等级/处理状态与英文枚举一一对应、无实质冲突后才动手。
